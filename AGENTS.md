# AGENTS.md

## 项目开发约束（TaskRoute）

### 1. 数据迁移管理规则

1. 开发阶段禁止新增“修补型迁移”来做历史结构调整。
2. 若表结构需要调整，直接修改已有迁移文件，不创建 `update_*`、`delete_*`、`alter_*` 等命名的迁移文件。
3. 迁移文件命名仅允许业务实体创建语义，例如：`create_xxx_table`。
4. 仅在项目进入生产且需要版本兼容时，才允许追加增量迁移。

### 1.1 API 设计规则

1. 禁止使用 RESTful API 风格（包括资源化路径、`apiResource` 路由约定）。
2. 统一使用动作式接口命名：`/xxx/list`、`/xxx/create`、`/xxx/detail`、`/xxx/update` 等。
3. 避免 URL 路径参数形式（如 `/{id}`），优先通过请求体传递业务参数。

### 2. 账号与权限规则

1. 系统不开放用户注册功能。
2. 所有账号由管理员在系统内创建、分配、启停和重置。
3. 产品按“终端形态”区分为 PC 与移动端，不按业务身份分站。
4. PC 与移动端使用同一账号体系与权限模型。
5. 管理员、调度员、司机都可使用移动端，移动端能力同样按权限控制。
6. 默认登录凭证使用“账号+密码”（非邮箱）。
7. 默认角色划分：
   - `admin`：全权限
   - `dispatcher`：调度相关权限
   - `driver`：仅个人任务权限

### 3. 登录安全规则

1. 登录页必须启用图片验证码。
2. 验证码失败时拒绝登录并提示重新获取。
3. 验证码需有过期时间且一次性使用。

### 4. 模拟数据要求

1. 开发环境必须提供可直接联调的模拟数据（用户、车辆、货品分类、禁混规则、预计划单）。
2. 模拟数据需覆盖油品与商超两个典型场景，便于验证禁混与派单策略。

### 5. Git 提交规则

1. 每一轮任务节点完成后，必须执行一次 Git 提交，不得长期堆积未提交改动。
2. 提交前必须完成该任务节点的基本可用性验证（至少通过相关测试、构建或关键流程自测）。
3. 提交信息需清晰描述该节点的核心变更，建议使用 `feat:`、`fix:`、`refactor:`、`test:` 等前缀。
4. Git 提交信息必须使用中文，不得使用英文提交说明；可保留 `feat:`、`fix:`、`refactor:`、`test:`、`chore:` 等前缀，但前缀后的变更描述必须为中文。

### 6. 任务推进规则

1. 每一轮任务节点完成后，必须明确给出下一步建议。
2. 在未收到用户“暂停”或“不继续执行”的前提下，给出下一步建议后应自动继续执行下一项最合理的任务。
3. 若下一步存在明显风险、较大范围改动或需要用户决策，先简要说明原因，再等待用户确认。
